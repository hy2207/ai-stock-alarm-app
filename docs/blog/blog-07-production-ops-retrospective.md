# 배포하고 나서가 진짜 시작이었다: 운영 버그, 프레임워크 업그레이드, 푸시 재활성화 기록

00~06편에서 시장조사부터 정량 모델 회고까지 정리했는데, 시리즈를 마무리하고 나서도 프로젝트는 계속 움직였다. 이번 편은 그 이후의 기록이다 — 실사용 중 발견된 무한 로딩 버그, 묵혀둔 레거시 청산, Next.js 15 / React 19 / Auth.js v5 업그레이드, 그리고 꺼둔 OneSignal 푸시를 다시 켜면서 겪은 삽질까지. 깔끔한 설계 얘기보다는 "켜보니 안 되더라"의 연속이라, 오히려 운영이 어떤 것인지 그대로 보여주는 기록이 됐다.

## 1. "생성 중입니다"에서 멈추는 화면 — 타임아웃 예산이 처음부터 안 맞았다

어느 날 홈에 들어갔더니 추천 카드 생성이 "생성이 오래 걸리고 있어요"에서 영원히 멈춰 있었다. 새로고침해도 그대로였다.

원인을 파보니 구조적으로 성공할 수 없는 시간 예산이었다. 관심종목 최대 3개를 **순차로** 처리하는데, 종목 하나당 LLM 호출이 재시도 포함 최대 50초까지 걸릴 수 있었다. 3종목이면 이론상 150초가 필요한데, 서버 함수 제한도 클라이언트 타임아웃도 전부 55~60초로 묶여 있었다. 종목이 2개 이상이면 매번 필연적으로 타임아웃이었던 것이다.

더 큰 문제는 그다음이었다. 타임아웃 응답을 보낸 뒤 "서버에서 계속 진행 중"이라고 안내했는데, 실제로는 Vercel이 응답을 반환하는 순간 함수를 정지시켜서 진행 중이던 생성 작업이 카드를 저장하지 못한 채 죽고 있었다. 안내 문구가 거짓말이었던 셈이다.

고친 방법은 세 가지다. LLM 호출을 순차에서 **병렬로** 바꿔서 최악의 경우를 종목 1개 기준(~50초)으로 줄였고, 함수 제한을 300초로 늘렸고, 응답 후에도 작업이 계속되도록 `waitUntil()`로 명시했다. 외부 API(Yahoo/Finnhub) 호출에 타임아웃이 하나도 없던 것도 이때 발견해서 전부 10초로 묶었다. 덤으로 단계별 소요시간을 로그로 남기게 해서, 다음에 느려지면 어느 단계가 병목인지 바로 보이게 했다.

## 2. 신규 상장주는 왜 예측이 안 뜨는가 — 침묵하는 빈 화면의 비용

SpaceX(SPCX)를 관심종목에 추가했더니 가격 차트에 예측 수치가 안 떴다. 버그인가 싶어 파봤는데, 상장한 지 19거래일밖에 안 된 종목이라 예측 모듈의 최소 기준(20거래일)에 딱 하루 모자란 것이었다. 설계된 동작이었다.

문제는 화면이 **아무 설명 없이 그냥 비어 있었다**는 점이다. 사용자 입장에서는 "왜 안 나오지?"가 될 수밖에 없다. 그래서 데이터가 부족할 때는 "거래 데이터가 19거래일 쌓였어요. 통계 예측은 최소 20거래일 데이터가 필요해서..."라는 안내와 진행률 바를 보여주도록 바꿨다. 조용히 생략하는 것과 이유를 말해주는 것의 차이는 생각보다 크다 — 이 빈 화면 하나 때문에 원인 조사에 시간을 쓴 게 바로 나였으니까.

## 3. 스캐폴드의 유령 — 의존성 43개와 컴포넌트 43개를 지웠다

이 프로젝트는 Figma Make/Vite 스캐폴드 위에 Next.js를 얹으면서 시작됐는데, 그 잔재가 생각보다 깊게 남아 있었다. package.json 이름이 여전히 `@figma/my-make-file`이었고, MUI·react-router·react-dnd처럼 **한 번도 import된 적 없는 패키지가 수두룩**했다. shadcn/ui 컴포넌트는 50개가 깔려 있는데 실제로 쓰는 건 7개였다.

전수조사 방법은 단순했다. 소스 전체에서 실제 import되는 패키지 목록을 뽑아서 package.json과 대조했다. 결과: 의존성 43개 제거, 죽은 UI 컴포넌트 43개 삭제, 총 8천 줄 가까이 줄었다. react가 dependencies가 아니라 optional peerDependencies로 선언돼 있던 비정상 구조도 이때 바로잡았다.

같은 김에 실패한 채 방치되던 테스트 13개도 청산했다. 전부 "코드는 바뀌었는데 테스트만 옛날 계약을 검증하던" stale 테스트였다 — 페이지 경로가 `/`에서 `/today`로 바뀐 것, Gemini 기본 모델명이 바뀐 것 같은 것들. 스위트가 항상 빨간 상태면 진짜 회귀가 묻힌다. 493개 전부 그린으로 만들고 나니 이후 작업마다 "내가 뭘 깨뜨렸는지"가 즉시 보였다.

## 4. Next 15 + React 19 + Auth.js v5 — 프리뷰에서 됐다고 프로덕션에서 되는 게 아니다

메이저 업그레이드는 별도 브랜치에서 진행했다. Next.js 공식 코드모드가 params/searchParams의 Promise 전환을 대부분 처리해줘서 프레임워크 업그레이드 자체는 순탄했다. NextAuth v4 → Auth.js v5도 `getCurrentUserId` 시그니처를 유지한 덕에 호출부와 테스트 mock을 전혀 건드리지 않고 끝났다. 프리뷰 배포에서 로그인까지 확인하고 머지했다.

그런데 프로덕션에서 로그인이 안 됐다. 로그를 보니 Google 콜백까지는 성공하는데, 로그인 직후 미들웨어가 다시 로그인 페이지로 튕겨내고 있었다.

원인은 v4와 v5의 미묘한 차이였다. v5의 `getToken`은 v4와 달리 `secureCookie`를 자동 감지하지 않고 **기본값 false로 둔다**. HTTPS 프로덕션에서 실제 쿠키 이름은 `__Secure-authjs.session-token`인데 미들웨어는 `authjs.session-token`을 찾고 있었던 것이다. 로그인도, 쿠키 발급도 전부 정상이었는데 미들웨어만 못 읽었다. node_modules에서 v5 구현체를 직접 열어보고서야 확정할 수 있었다 — 마이그레이션 가이드만 믿지 말고, 안 맞으면 라이브러리 소스를 열어보는 게 결국 제일 빠르다.

## 5. 푸시 재활성화 — 마커 6곳을 복원했는데 7번째가 숨어 있었다

마지막은 꺼두었던 OneSignal 웹 푸시 재활성화. 처음 비활성화할 때 `PUSH_DISABLED` 마커를 남겨둔 덕에 어디를 복원해야 하는지는 명확했다. 그런데 이 작업이 삽질의 연속이 됐다.

**켜기 전에 발견한 치명적 버그**: 크론은 external_id(우리 DB의 유저 ID)로 발송하는데, 클라이언트 어디에서도 `OneSignal.login(userId)`을 호출한 적이 없었다. 이대로 켰으면 발송은 매일 "성공"하는데 수신자는 영원히 0명이었을 것이다. 발송 로직과 구독 로직이 각자 완성돼 있어도, 그 둘을 잇는 연결 고리는 따로 검증해야 한다.

**숨어 있던 7번째 마커**: `PUSH_DISABLED` 마커를 grep해서 6곳을 복원했는데, 구독이 계속 "no active Service Worker"로 실패했다. 알고 보니 `public/OneSignalSDKWorker.js`가 여전히 "설치되자마자 스스로 등록 해제하는" 비활성화 스텁이었다. 검색 범위를 `src/`로 좁힌 대가였다. 마커 검색은 반드시 저장소 전체에서.

**CSP 두더지잡기**: OneSignal SDK v16은 스크립트를 CDN에서만 받는 게 아니었다. sync 엔드포인트를 JSONP 방식(script 태그)으로 부르고, 프롬프트 UI용 스타일시트도 자기 도메인에서 불러온다. `connect-src`만 열어서는 부족하고 `script-src`, `style-src`까지 세 번에 걸쳐 CSP를 열어야 했다. 콘솔의 CSP 위반 메시지를 하나 잡으면 다음 것이 나타나는 식이라, 서드파티 SDK를 CSP 환경에 들일 때는 필요한 지시어 목록을 먼저 문서에서 확인하는 게 맞다.

**마지막 반전, API 키**: 코드를 다 고치고도 발송이 실패했다(`sent: 0, failed: 1`). 로컬에서 같은 자격증명으로 OneSignal API를 직접 호출해보니 403 — 환경변수에 저장돼 있던 REST API 키가 애초에 유효한 형식이 아니었다(정상 키는 `os_v2_`로 시작하는 113자, 저장된 건 25자). 키를 재발급받아 교체하자 그제서야 `scheduled: 1, sent: 1, failed: 0`.

그리고 알림이 안 보여서 마지막으로 확인한 곳은 코드가 아니라 **macOS 시스템 설정**이었다. Chrome 알림 허용이 꺼져 있었다. 서버, CSP, 서비스워커, API 키를 다 통과한 알림이 OS 설정 하나에 막힐 수 있다는 것까지가 이 파이프라인의 전체 길이다. 실제 단말 수신의 최종 확인은 아직 진행 중이다 — 이 글도 그래서 "완료 보고"가 아니라 중간 기록이다.

## 6. 정리

이번 사이클에서 남은 교훈을 추리면 이렇다.

1. **타임아웃 예산은 합산으로 검증한다.** 각 단계가 그럴듯해도 합치면 처음부터 불가능한 예산일 수 있다.
2. **"백그라운드에서 계속 진행"은 공짜가 아니다.** serverless에서 응답 후 실행은 명시적으로(`waitUntil`) 보장해야 한다.
3. **조용한 빈 화면은 버그 리포트를 만든다.** 의도된 생략에는 이유를 표시하자.
4. **stale 테스트는 빚이다.** 스위트가 빨간 채로 두면 진짜 회귀를 놓친다.
5. **프리뷰 통과 ≠ 프로덕션 통과.** 특히 쿠키/도메인이 얽힌 인증은 프로덕션에서 한 번 더.
6. **기능 플래그 마커는 저장소 전체에서 검색한다.** `src/` 밖에도 코드가 산다.
7. **발송 로직과 구독 로직 사이의 "연결"은 별도의 검증 대상이다.** 양쪽이 각자 완벽해도 이어져 있지 않으면 수신자는 0명이다.

00~06편이 "어떻게 만들었나"였다면, 이번 편은 "만든 다음에 무슨 일이 생기나"에 가깝다. 그리고 운영은 아직 끝나지 않았다 — 내일 아침 7시, 첫 자동 브리핑 푸시가 예정되어 있다.
