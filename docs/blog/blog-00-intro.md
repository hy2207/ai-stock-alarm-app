# 3개월짜리 개인 프로젝트를 시작한 이유: 프롬프트 한 줄로 만든 웹앱 말고

지난 3월부터 6월까지, 모두의연구소에서 진행하는 기획/개발 관련 수업을 들으면서 개인 프로젝트를 하나 진행했다. 미국주식 관심종목을 매일 아침 1~3개의 추천 카드로 압축해주는 "AI 주식 알람 앱"이다. 이번 글부터 이어지는 시리즈는 이 3개월 동안의 기록을 정리한 것이다.

## 1. 왜 시작했나

AI가 빠르게 좋아지면서, 프롬프트 몇 줄로 웹앱을 뚝딱 만들어보는 경험은 이미 여러 번 해봤다. 그런데 그렇게 만든 결과물들을 돌아보면 항상 비슷한 한계가 있었다 — 화면은 그럴싸하게 나오는데, "이 기능을 왜 만들었는지", "이 다음엔 뭘 만들어야 하는지"에 대한 답이 없었다. 프롬프트가 잘 만들어준 건 결과물이지, 기획이 아니었다.

그래서 이번엔 반대로 해보기로 했다. 프롬프트로 빨리 만드는 대신, 실제 필드에서 기획과 개발이 어떤 순서로 진행되는지를 처음부터 끝까지 직접 밟아보기로 한 것이다. 시장조사 → Pain/KPI 정의 → PRD → SRS → 태스크 분해 → 실제 구현 → 검증/회고까지, 어느 하나도 건너뛰지 않고 문서로 남기면서 진행했다.

## 2. 이번 프로젝트로 얻은 것

3개월을 지나고 나서 가장 크게 남은 건 두 가지다.

하나는, AI에게 시키는 일의 품질이 결국 내가 먼저 정리해둔 문서의 품질을 못 넘는다는 것이다. PRD와 SRS를 제대로 못 써놓으면 아무리 좋은 모델을 붙여도 매번 다른 결과물이 나온다. 반대로 Pain, KPI, ADR, REQ ID를 표로 못박아두면, 어떤 AI 도구를 켜도 같은 방향으로 개발이 진행된다는 걸 몸으로 확인했다.

다른 하나는, 기획이 한 번 끝나는 게 아니라는 것이다. 통계 예측 모델 하나를 실제로 세 번 갈아엎으면서, 문서로 미리 다 정할 수 있는 영역과 데이터를 쌓아가며 계속 고쳐야 하는 영역이 따로 있다는 걸 배웠다. 이 부분은 시리즈 마지막 편에서 자세히 다룬다.

## 3. 시리즈 계획

앞으로 이어질 글은 실제 진행 순서를 그대로 따라간다.

1. **시장조사 없이 시작한 기능 목록은 왜 산으로 가는가** — Pain 지표로 PRD 자르기
2. **PRD를 AI 에이전트가 읽게 만들기** — SRS를 따로 쓰는 이유
3. **SRS 한 줄이 커밋 20개가 되기까지** — 요구사항을 태스크로 쪼개는 스킬
4. **하나의 규칙, 세 개의 AI 도구** — Cursor·Claude Code·Antigravity 동시 운영기
5. **서브에이전트 6개로 나눈 이유** — 하나로 뭉쳐두지 않은 기준
6. **예측 모델을 두 번 갈아엎은 이유** — Conformal Prediction과 Yang-Zhang 변동성 도입기

1~2편은 기획 단계, 3~5편은 이 기획을 AI 도구와 함께 실제 코드로 옮기는 과정, 6편은 그렇게 나온 결과물 중 하나를 실측 데이터로 검증하고 고친 회고다. 순서대로 읽으면 기획 문서 한 줄이 실제로 어떤 코드와 프로세스로 이어지는지 흐름이 보이게끔 구성했다.
